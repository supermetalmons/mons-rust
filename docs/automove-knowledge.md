# Automove Knowledge

This file keeps durable automove rules and reusable heuristics only.

Use `HOW_TO_ITERATE_ON_AUTOMOVE.md` for the workflow, `AUTOMOVE_IDEAS.md` for the live state, and `docs/automove-archive.md` for retired wave detail.

## Stable Runtime Truths

- Public Pro routes through `frontier_pro_v2_guarded`.
- `shipping_pro_search` is the retained search-only baseline.
- Release wiring is intentionally narrower than the experiment surface: public `Pro` dispatch goes through `MonsGameModel::public_runtime_inputs` to `select_frontier_pro_v2_guarded_inputs`, while `automove_experiments` and experiment profile selectors are only included under `#[cfg(test)]`.
- Probe paths are diagnostics only; they do not describe shipping behavior.
- Promotion evidence comes from direct frontier-vs-baseline duels, not fixture churn alone.
- `runtime-preflight` still matters after promotion: exact-lite is hard, stage-1 CPU is advisory for Pro.
- Automove users can play any current `GameVariant`; promotion evidence must cover variant breadth, not just Classic.

## Experiment Rules

- Pick one hypothesis before editing runtime code.
- Probe first when the mechanism is unclear. Do not spend canonical gates on a guess.
- When the live board has no challenger, use the structural reset in `docs/automove-strategy.md`; do not continue the seam-fix loop by default.
- For reset-mode work after 2026-04-27, read `docs/automove-major-reset-plan.md` first. The next source-bearing implementation should be outcome-corpus V2, a test-only ProV4 unified root policy, or a corpus-calibrated utility feature; another selector over policy labels, exact contexts, branches, or first moves is not a valid default.
- Run `./scripts/run-automove-structural-scout.sh <candidate>` before retaining any broad Pro runtime change. A candidate must look strong on both canonical sampled and active-blocker panels, not just one.
- Treat retained Classic fixtures as regression controls, not proof that a candidate is stronger across variants.
- Use seeded sampled variants for quick kill/pass evidence, then all-variant confirmation for promotion.
- Separate `pre_accept` search choice from final `engine_post_search` output before changing shared heuristics.
- A seam can move while the duel gate still fails; local seam coverage is not duel strength.
- Use `SMART_PRO_RELIABILITY_HOTSPOT_FEN` on `smart_automove_pro_reliability_hotspot_probe` for one-off board inspection. Do not add temporary source cases just to inspect a copied trace board.
- Use `SMART_PRO_RELIABILITY_DUEL_FILTER` on `smart_automove_pro_reliability_duel_trace_probe` or `smart_automove_pro_reliability_nonwin_trace_probe` when only one duel bucket needs recurrence evidence.
- Use `smart_automove_pro_profile_sweep_probe` when the live surface is too mixed for another seam patch. The probe compares test-only Pro candidates without adding them to the retained profile registry; its output is discovery evidence, not promotion evidence.
- Prefer the `run-automove-experiment.sh` diagnostic stages for repeatable profile comparisons: `pro-profile-sweep` wraps `smart_automove_pro_profile_sweep_probe`, and `pro-profile-attribution` wraps `smart_automove_pro_profile_attribution_probe`. These stages are diagnostics only; retained-profile restrictions still apply to canonical promotion stages.
- Use `pro-promotion-dashboard` before spending runtime code on a broad Pro candidate. It reports sampled plus active-blocker shape, weakest variant rows, candidate-vs-shipping strength, and candidate-vs-guarded pressure in one run; profile attribution should follow only when the dashboard leaves a globally promising candidate with one explainable miss. For high-risk or expensive scouts, first run `SMART_PRO_DASHBOARD_PANEL_FILTER=sampled SMART_PRO_DASHBOARD_INCLUDE_GUARDED=false` and kill immediately on a sampled miss.
- Read `PRO_PROMOTION_DASHBOARD_STOPLIGHT` before inspecting long dashboard output. `promotable_shape` earns confirm spend, `sampled_only` and `active_only` require decision-record proof before more source work, `broad_pressure` is diagnostic pressure rather than promotion evidence, `cost_blocked` kills or narrows the line, and `not_promising` means discard the candidate unless it exposed a harness issue.
- Structural scout defaults to dashboard fast-fail. Set `SMART_PRO_DASHBOARD_PROMOTION_FAST_FAIL=false` only when a full sampled plus active dashboard is the explicit diagnostic target.
- Config-toggle sweep candidates should preserve the public `frontier_pro_v2_guarded` wrapper unless the candidate is explicitly a raw-control profile. Otherwise the dashboard mostly measures raw frontier plus a tweak, not public Pro plus the tweak. `frontier_pro_v2_raw` remains the explicit raw surface; runtime-toggle candidates should route their modified frontier runtime through the guarded wrapper helper.
- Read `AUTOMOVE_SWEEP_CANDIDATE_METADATA` before sweep, dashboard, policy-matrix, policy-corpus, outcome-corpus, cross-budget, winner, or attribution logs. The runner prints each candidate's mechanism, expected invariant, risk rows, and kill condition so stale ablations stay diagnostic and do not reappear as direct source work.
- Current ProV4 root-pool families are archived as no-source unless a future corpus introduces a genuinely new measured field. Do not build runtime selectors from current root-pool fields, guarded deltas, exact-pressure posture, coarse resource/outcome/legal/attack/support/territory/mana/consumable/engagement/mobility/action-threat/role/base/lane/transition/worst-reply/reply-spectrum features, policy labels, branch labels, first moves, or singleton-heavy corpus evidence.
- Root-breadth config rebalancing is also not a direct ProV4 path. A guarded wrapper candidate with root focus `4`, root focus share `6_500`, turn-engine seed/per-family/opponent/reply breadth `16 / 5 / 7 / 4`, and expansion cap `184` cleared a tiny Fast smoke, then failed sampled Pro/Normal at `9-3 / 8-4`; do not reopen root-focus or seed-cap breadth without a new corpus-discovered root feature.
- Targeted exact-turn memo plus score-window candidate narrowing is not a direct ProV4 path either. The guarded wrapper candidate cleared a tiny Fast smoke but failed sampled Pro/Normal at `7-5 / 7-5`, including Pro `inner_wedge_mana_rows=0-2`; do not reopen targeted exact narrowing without corpus evidence that it separates candidate wins from guarded saves.
- Child eval bundles plus two-stage child ordering are not enough as a direct ProV4 config mode. The guarded wrapper candidate reduced sampled Pro average time to `106.52ms` but still failed sampled Pro/Normal at `8-4 / 8-4`, with Normal `alternating_mana_rows=0-2`; do not reopen child-ordering shortlist tuning without a corpus signal that the saved cost does not drop quiet saves.
- PVS plus killer/history/TT search ordering is not enough as a direct ProV4 config mode. The guarded wrapper candidate stayed under the move-time ceiling but failed sampled Pro/Normal at `6-6 / 9-3`, with Pro `inner_wedge_mana_rows=0-2`; do not reopen traversal-order toggles without a corpus signal that they preserve quiet saves.
- Root aspiration windowing is not enough as a direct ProV4 config mode. A guarded wrapper candidate that restored `enable_root_aspiration` cleared a mechanical tiny Fast smoke but did not meet the sweep pass threshold (`duel_passes=false`, `217.54ms` average), then sampled Pro fast-failed at `5-7` as `not_promising` with `inner_wedge_mana_rows=0-2` and `169.68ms` average; do not reopen root-window toggles without corpus evidence that they improve selection while preserving quiet saves.
- Drainer walk-threat safety filtering is not enough as a direct ProV4 config mode. A guarded wrapper candidate that enabled `enable_walk_threat_prefilter` cleared a mechanical tiny Fast smoke but did not meet the sweep pass threshold (`duel_passes=false`, `209.81ms` average), then sampled Pro fast-failed at `7-5` as `not_promising` with `inner_wedge_mana_rows=0-2` and `143.94ms` average; do not reopen walk-threat root filters without corpus evidence that the safety signal separates saves from quiet-progress losses.
- Disabling tactical quiescence is not enough as a direct ProV4 config mode. A guarded wrapper candidate with `enable_quiescence_search=false` cleared a mechanical tiny Fast smoke but did not meet the sweep pass threshold (`duel_passes=false`, `166.49ms` average), then sampled Pro fast-failed at `8-4` as `not_promising` with `inner_wedge_mana_rows=0-2` and `154.89ms` average; do not reopen quiescence toggles without corpus evidence that leaf tactical probes separate wins from quiet-save losses.
- Restoring forced tactical prepass is not enough as a direct ProV4 config mode. A guarded wrapper candidate with `enable_forced_tactical_prepass=true` cleared a mechanical tiny Fast smoke but did not meet the sweep pass threshold (`duel_passes=false`, `195.01ms` average), then sampled Pro fast-failed at `6-6` as `not_promising` with `alternating_mana_rows=0-2` and `121.98ms` average; do not reopen tactical-prepass toggles without corpus evidence that prepass roots separate tactical saves from quiet-save losses.
- Disabling futility pruning is not enough as a direct ProV4 config mode. A guarded wrapper candidate with `enable_futility_pruning=false` cleared a mechanical tiny Fast smoke but did not meet the sweep pass threshold (`duel_passes=false`, `233.58ms` average), then sampled Pro fast-failed at `8-4` as `not_promising` with `inner_wedge_mana_rows=0-2` and `171.62ms` average; do not reopen futility-pruning toggles without corpus evidence that pruned leaves separate wins from quiet-save losses.
- Shallow-vs-full root depth consensus is not enough as a direct ProV4 selector. A guarded wrapper candidate overrode only `frontier_execute` roots and stayed under the move-time ceiling, but sampled Pro failed at `6-6`, with `inner_wedge_mana_rows=0-2` and `forward_bridge_mana_rows=1-3`; do not reopen depth-stability root overrides without corpus evidence that depth-instability separates candidate wins from guarded saves.
- Forced drainer-attack pressure is not enough as direct ProV4 config tuning. Score-aware attack forcing and minimax selection inside forced attack roots each stayed below the move-time ceiling but failed sampled Pro at `7-5`, with `inner_wedge_mana_rows=0-2`; do not reopen attack-forcing toggles without corpus evidence below drainer-pressure labels.
- When self-scouting the shipped `frontier_pro_v2_guarded` profile in structural-reset mode, skip the guarded-delta self-comparison. `scripts/run-automove-structural-scout.sh` now does this by default for that exact candidate; force `SMART_PRO_DASHBOARD_INCLUDE_GUARDED=true` only when the redundant self-delta is intentionally needed.
- Use `pro-policy-matrix` when candidate policies are complementary on isolated active rows but suspect across sampled or mode panels. The first candidate is the baseline; the matrix replays identical openings and reports candidate-vs-baseline outcome deltas, nonwins, first-move divergences, and branch/context/pair aggregates.
- `pro-policy-outcome-corpus` mechanism classes now include `axis=exact_pressure` and `axis=exact_timing`, and the portfolio stoplight can emit `repeated_mechanism_class` when those or other mechanism-class axes repeat above the known-bad count-2 threshold. Use these axes to separate zero-window repairs from weak-window saves before designing another ProV4/root-origin selector.
- Use `pro-policy-cross-budget` before turning policy-winner data into a selector. It replays the same opening side against Pro, Normal, and Fast shipping opponents and reports whether one policy is an all-budget win, a non-regressing repair, a budget conflict, or no help. A `budget_conflict` means a policy helps one opponent budget while regressing another on the same opening side, so the next spend needs a shared utility/root-evaluation feature rather than another turn/color/context gate.
- For active-blocker cross-budget follow-ups, set `SMART_PRO_POLICY_CROSS_BUDGET_GAMES=3`; the cross-budget probe's default `1x1` shape only covers the first active opening and can miss alternating/forward-bridge evidence. The 2026-04-27 `frontier_pro_v3_full_scored_reply_guard` active follow-up stayed `partial_repair_coverage_gap` after this widening, so alternating-only full-scored routing is not a source path.
- Use `SMART_PRO_POLICY_CROSS_BUDGET_SEED_OPPONENT_MODE=pro|normal|fast` plus a matching `SMART_PRO_POLICY_CROSS_BUDGET_SEED_TAG` when the cross-budget check must replay the exact seed family from a policy-winner duel. Copy `seed_tag` from `PRO_POLICY_WINNER_SUMMARY` or `PRO_POLICY_WINNER_STOPLIGHT`; for example, active blockers vs Fast uses `pro_profile_active_blockers_v1_vs_fast`, not the unsuffixed panel seed. Use `SMART_PRO_POLICY_CROSS_BUDGET_STATE_LIMIT` to keep these follow-ups to the same first few states.
- Use `SMART_PRO_POLICY_CROSS_BUDGET_INCLUDE_MECHANISM_CLASS=true` after a narrow cross-budget run finds stable repairs and you need the root/advisor/utility mechanism below the policy label. The default mechanism filter is `stable`, so it classifies only clean all-budget and non-regressing repairs; switch to `conflicts` or `all` only for a focused save-risk follow-up. Watch `PRO_POLICY_CROSS_BUDGET_STOPLIGHT`: `mixed_cross_budget` means a clean-looking policy also has budget conflicts in the same slice, and `max_stable_mechanism_class_states <= 2` is still routing evidence only.
- Cross-budget stoplights now surface `coverage_gap`, `partial_repair_coverage_gap`, and `baseline_save_risk` before repair labels. Treat those as source-work kills: they mean the checked policy family either cannot cover enough same-opening budget outcomes or loses openings that guarded already wins across all budgets.
- A single `pro-policy-cross-budget` non-regressing repair is not enough to scope a policy by variant. Full-scored reply guard repaired the first sampled `inner_wedge_mana_rows` opening without cross-budget regression, but an `inner_wedge`-scoped full-scored candidate still failed sampled Pro at `7-5` and made that row `0-2`.
- The 2026-04-27 cross-budget mechanism smoke did not make full-scored reply guard promotable. Sampled `inner_wedge_mana_rows` white was a one-state non-regressing repair with safe `ManaTempo` safe-step progress losing to safe `SpiritImpact` development, but the active-blocker four-state slice had only one clean all-budget outer-edge repair, two budget conflicts, one baseline-only all-budget save, and `max_stable_mechanism_class_states=1`. Do not retry full-scored or safe-spirit selection from this evidence without a repeated cross-budget-stable class.
- The same 2026-04-27 conflict-class follow-up killed safe-step/spirit reply-guard filtering more directly. Active `alternating_mana_rows` white had a `budget_conflict` with the same safe `ManaTempo` safe-step to safe `SpiritImpact` development class as the sampled non-regressing repair, while a narrow opposite-direction full-scored candidate (`SpiritImpact` development to `ManaTempo` safe-step) failed sampled Pro at `6-6` with `inner_wedge_mana_rows=0-2` and `forward_bridge_mana_rows=1-3`. Do not retry bidirectional safe-step/spirit routing over full-scored reply guard.
- The same 2026-04-27 reset follow-up found no promotable shipping-control class. A sampled four-state `shipping_pro_search_control` check was `mixed_cross_budget` with one clean black inner-wedge repair, one center-spoke budget conflict, and one no-help state. The active four-state check had one non-regressing outer-edge repair, two no-help states, a baseline-only all-budget save, and no repeated stable mechanism. Do not retry shipping-control routing from these singleton classes.
- A sampled Normal outcome-corpus `repeated_mechanism_class` can still be a false positive. On the first four sampled Normal states, the reset portfolio had `candidate_only_wins=2`, `baseline_only_wins=0`, and `max_mechanism_class_games=3`, but exact seed-aligned cross-budget over the repairing family was `mixed_cross_budget`: `frontier_pro_v2_no_selected_followup_projection` cleanly repaired inner-wedge white across all budgets, while corner-chain black was a budget conflict where shipping/raw/full-scored each improved one budget and regressed another. Do not encode the Normal class without a feature that separates that corner-chain conflict.
- A sampled Fast outcome-corpus slice can also be pure guarded save evidence. On the first four sampled Fast states, guarded won every state and the portfolio stoplight was `shared_only`; shipping-control, raw, no-selected-followup, and full-scored had only regression-only or no-delta candidate stoplights. Treat sampled Fast shared-only slices as save-risk evidence, not selector upside.
- A capped sampled Pro outcome-corpus slice can be pure guarded save evidence too. On the first two sampled Pro states, guarded won both checked `inner_wedge_mana_rows` states and the portfolio stoplight was `shared_only`; shipping-control, raw, no-selected-followup, and full-scored were regression-only or no-delta, with repeated baseline-better mechanism pressure in zero-window safe contexts. Do not reopen the existing-policy sampled corpus route unless Outcome Corpus V2 adds a feature below these policy labels.
- Use `pro-policy-winner` after a policy matrix has good oracle coverage and you need selector design data faster. It short-circuits candidate replay after the first winning policy for each baseline loss and aggregates the winner policy, exact context, and move pair.
- `pro-policy-winner` also emits `PRO_POLICY_WINNER_GLOBAL_*` aggregates across the selected panels and duels. Use these global lines to check whether a mechanism repeats across filtered slices before reading long per-duel output. Mechanism-enabled winner runs are expensive; keep `SMART_PRO_POLICY_WINNER_PANEL_FILTER`, `SMART_PRO_POLICY_WINNER_DUEL_FILTER`, `SMART_PRO_POLICY_WINNER_REPEATS`, and `SMART_PRO_POLICY_WINNER_GAMES` narrow until a small slice proves useful.
- Use `./scripts/run-automove-structural-scout.sh --corpus <candidate>` when there is no live runtime hypothesis. It runs the promotion dashboard and a bounded default policy corpus portfolio, giving future work both promotion-shape evidence and root/advisor/head/utility mechanism stoplights before source edits. The scout defaults to `SMART_PRO_POLICY_WINNER_STATE_LIMIT=2` and `SMART_PRO_POLICY_WINNER_CANDIDATE_TRACE_LIMIT=64`; widen only after the bounded corpus finds a repeated mechanism worth validating.
- Broad `pro-policy-winner` portfolios are still expensive. Prefer `SMART_PRO_POLICY_WINNER_PANEL_FILTER`, `SMART_PRO_POLICY_WINNER_DUEL_FILTER`, `SMART_PRO_POLICY_WINNER_STATE_LIMIT`, and `SMART_PRO_POLICY_WINNER_CANDIDATE_TRACE_LIMIT` when refreshing coverage; treat `state_limit_hit=true` or `candidate_trace_limit_hit=true` summaries as partial diagnostic evidence only.
- A full default policy corpus can show oracle coverage without selector viability. The 2026-04-26 reset corpus had `no_policy_wins=0` across sampled and active-blocker Pro/Normal/Fast duels, but every `PRO_POLICY_WINNER_POLICY` and `PRO_POLICY_WINNER_MECHANISM` aggregate stayed at `games=1`; that means the next spend should be an outcome-corpus stoplight, ProV4 unified root policy, or a new measured utility feature, not a static selector over the existing policy labels.
- Use `PRO_POLICY_WINNER_MECHANISM_CLASS` before concluding that exact mechanism records are true singleton residue. The class output intentionally drops exact board, variant, policy, and move-pair detail and groups root status by stage/head state, role, family, advisor status, rank, safety/progress, and winner-root shape. Treat repeated classes as routing evidence for the next focused probe, not as runtime permission by themselves.
- Mechanism-class output includes `axis=rank_score_delta`, a coarse winner-vs-baseline rank and score-gap bucket. This is a corpus-side feature for discovering repeated root-evaluation shapes; do not recompute root utility or cross-budget evaluation inside a sweep selector. The first sampled Normal full-slice validation after adding this axis still ended as `singleton_residue`: rank/score-delta buckets were all singleton, and the only repeated classes were broad role/safety buckets.
- Sampled Fast gives the same warning: the full sampled Fast winner slice had complete oracle coverage (`6` baseline wins and `6` policy wins) but `singleton_residue`; rank/score-delta buckets stayed singleton and only a broad safe-step safety/progress class repeated at count `2`.
- Read `PRO_POLICY_WINNER_STOPLIGHT` before designing selectors from a policy corpus. `singleton_residue` kills selectors over the current policy labels, `coverage_gap` means add a new policy/root feature first, `repeated_mechanism` is the only label that earns deeper runtime design, and `partial_corpus` means rerun narrower or raise the trace cap before making source decisions.
- Treat `repeated_mechanism_class` as a routing label, not runtime permission. The winner stoplight only emits it above the known-bad count-2 threshold; it should trigger a narrower cross-budget probe or utility-feature investigation, not a direct selector over the class.
- The active Fast `no_selected_followup_projection` repair is not promotable from count-2 mechanism evidence. A seed-aligned active Fast cross-budget replay split the same lower-live safe-step `ManaTempo` hint into one clean white repair and one black `budget_conflict`; do not turn that class or policy label into source code without a new shared utility/root-evaluation feature.
- Active Fast shipping-control repairs are also split by budget even when the first-winning policy is repeated in a tiny slice. A two-state exact-seed active Fast check found `shipping_pro_search_control` was a white `budget_conflict` on `outer_edge_mana_rows` and only a black non-regressing repair; the scoped black-only test candidate reached only active Fast `3-3`, and stacking it with the known sampled opening components was cost-blocked before a sampled dashboard result. Do not retry outer-edge action+mana shipping-control routing without a new repeated cross-budget feature.
- Broad `pro-policy-cross-budget` portfolios are more expensive than single-duel matrix runs. It defaults to sampled `1x1`; keep that shape and the smallest candidate set that explains a conflict first, then widen only after the narrow summary shows clean repairs.
- Treat `pro-policy-winner` output as selector evidence, not promotion evidence. The first winning policy can be a later manifestation of a policy that needed to enter earlier, and the same exact-opportunity context can prefer different policies across Pro/Normal/Fast; always validate any selector on `pro-promotion-dashboard`.
- Sampled Pro policy-winner coverage alone is a false positive. A selector that patched the current sampled Pro winner contexts reached `11-1` against shipping Pro, then failed sampled Normal at `8-4` and sampled Fast at `6-6`. The later `corner_chain_mana_rows` white `no_policy_win` was a forced-root oracle harness bug, not a root-pool ceiling, but fixing that coverage did not make static selectors promotable.
- Sampled-Pro opening plus utility composites are still false positives when their lower-budget residue is singleton. A test-only spirit/rank utility selector reached sampled Pro `10-2`, and adding exact early-white opening winners reached sampled Pro `11-1`, but the same candidate failed sampled Normal/Fast at `6-6 / 6-6`; Normal's six policy-winner repairs were all singleton by policy, context, and move pair. Kill this shape unless a cross-budget corpus produces repeated mechanisms below the policy labels.
- Full rollout policy selectors are not a practical escape hatch. A test-only portfolio rollout from early `frontier_execute` boards with only 24 internal plies did not complete a sampled Pro quick check after several minutes, and a tightened two-ply Pro/Fast continuation version was still cost-blocked. Future rollout ideas need a much cheaper precomputed feature extraction path before they can be promotion candidates.
- Shallow ProV4 unified root-value comparators are not enough when they only re-sort scored roots by existing tactical, `TurnEngineUtility`, safety, progress, family, score, and rank fields. Test-only comparators improved sampled Pro from `7-5` to `9-3`, including an opening-gated version that fixed `alternating_mana_rows`, but the residue stayed split across center, inner-wedge, and forward-bridge. Decision records labeled the line `singleton_regression_pressure`; policy-winner repairs split one each across guarded, raw, and shipping-control with no repeated context. A future ProV4 attempt needs a new corpus-trained feature or repeated mechanism below the existing root fields, not another tuple ordering.
- Root-origin plus continuation-probe ProV4 comparators are the same shallow family unless they introduce a new discriminator. A test-only unified pool over guarded, head, legacy, and top roots with one allowed-head continuation probe repaired a sampled Pro `inner_wedge_mana_rows` black turn-six zero-window board, but regressed a sampled Pro `center_spoke_mana_rows` black turn-eight weak-window board; the slice stayed `2-2` overall and cost rose from guarded `130.08ms` to `206.82ms`. Do not retry root-origin or post-root continuation selectors until a corpus feature separates zero-window repairs from weak-window saves.
- Selector-time cross-budget utility admission is not currently practical. A test-only Pro/Normal/Fast primary-axis admission rule preserved guarded wrapper branches and narrowed to vulnerable nearby roots, but the sampled-only dashboard stayed cost-blocked before output; cross-budget utility work needs corpus-side or precomputed features before it can be a promotion candidate.
- Selector-time utility tiebreaking is also not practical when it reruns root scoring after guarded selection. A test-only low-rank/top-3 utility tiebreak remained too slow for both the sampled dashboard and a one-opening profile sweep, so future utility work should add reusable snapshots or corpus-side features first.
- The test harness now has an opt-in root-selection snapshot for that cheaper path: call `request_turn_engine_root_selection_snapshot()` before running a selector, then read `turn_engine_root_selection_snapshot()` to inspect the selected inputs plus scored roots without repeating root scoring. A cached utility/rank tiebreak using this path was fast enough for a micro sweep but still non-promotable (`1-1` in each Pro/Normal/Fast one-opening duel), so the snapshot is a harness capability, not evidence for another utility tiebreak.
- Snapshot-based advisor/top-root reranking is not a promotion path by itself. A test-only advisor reply-floor policy that reranked already-scored roots by primary utility, progress/setup value, and reply floor completed the sampled dashboard but failed Pro/Normal/Fast at `5-7 / 9-3 / 8-4`, with broad weak rows and more than `100` overrides per duel. Future snapshot use needs a corpus-trained discriminator, not another hand-tuned progress/reply-floor tuple.
- Capped sampled Normal/Fast policy-winner slices remain singleton-heavy. A 4-state sampled Normal slice found two one-off shipping-control SpiritImpact repairs with different advisor/rank roles, while sampled Fast found one omitted shipping-control root; neither is enough for source work.
- Use `smart_automove_pro_decision_record_aggregation_probe` before building a shipping-root or context meta-selector. Run `SMART_PRO_DECISION_RECORD_SCOPE=nonwins` when the missed promotion gate contains flat losses, because delta-only attribution hides games where both frontier and shipping lose.
- A decision-record refresh that says all nonwins are `frontier_execute` is still not a runtime hypothesis. Require recurrence below that branch label, such as the same variant/color/turn/resource shape plus advisor/head/final-selection mechanism, before writing another guard, head veto, utility override, or policy selector.
- Read `PRO_SWEEP_DECISION_RECORD_STOPLIGHT` after `pro-sweep-decision-record` on arbitrary sweep candidates. `repeated_pair` and `repeated_context` can justify attribution; `branch_only` and `branch_only_with_regressions` are too coarse for runtime code; `singleton_residue` and `singleton_regression_pressure` kill static selectors; `missing_first_diff` means the trace is incomplete.
- `pro-promotion-dashboard` defaults `SMART_PRO_DASHBOARD_SKIP_GUARDED_AFTER_SHIPPING_FAIL=true`; after a panel fails all shipping-direction gates it emits `PRO_PROMOTION_DASHBOARD_GUARDED_SKIPPED` instead of spending on guarded-delta duels. Override this only for explicit guarded-delta attribution after a candidate already looks directionally useful.
- Use `SMART_PRO_DASHBOARD_PROMOTION_FAST_FAIL=true` for broad retained-candidate scans; it emits `PRO_PROMOTION_DASHBOARD_FAST_FAIL` after the first failed sampled strict gate or active directional gate. This is a scan accelerator, not promotion evidence, and should be disabled when you need complete per-duel weakness summaries.
- Use `SMART_PRO_SWEEP_DECISION_RECORD_INCLUDE_MECHANISM_CLASS=true` when `pro-sweep-decision-record` reports singleton contexts but the branch-level story is still tempting. The run then emits `PRO_SWEEP_DECISION_RECORD_MECHANISM_CLASS` and adds `max_mechanism_class_games` to the stoplight, grouping recorded nonwins/regressions by stage/head state, role, family, advisor status, rank, safety/progress, and compared root shape. As with policy-winner classes, count-2 repeats are routing evidence only; require a stronger repeated class or cross-budget-stable utility feature before runtime code.
- The 2026-04-27 retained Normal/Fast refresh stayed non-promotable even with decision-record mechanism classes enabled. Retained guarded had Normal `5` nonwins / `4` regressions and Fast `6` / `4`; both duels had `max_context_games=1`, `max_pair_games=1`, and only `max_mechanism_class_games=2`. A sampled-Normal raw/no-selected outcome slice was also unsafe: seed-aligned cross-budget over guarded/raw/no-selected had `0` clean repairs, `1` nonregressing repair, and `2` budget conflicts. Do not turn raw early-white, no-selected-followup, or count-2 mechanism-class evidence into a selector.
