# Automove Knowledge

This file keeps durable automove rules and reusable heuristics only.

Use `HOW_TO_ITERATE_ON_AUTOMOVE.md` for the workflow, `AUTOMOVE_IDEAS.md` for the live state, and `docs/automove-archive.md` for retired wave detail.

## Stable Runtime Truths

- Public Pro routes through `frontier_pro_v2_guarded`.
- `shipping_pro_search` is the retained search-only baseline.
- Release wiring is intentionally narrower than the experiment surface: public `Pro` dispatch goes through `MonsGameModel::public_runtime_inputs` to `select_frontier_pro_v2_guarded_inputs`, while `automove_experiments` and experiment profile selectors are only included under `#[cfg(test)]`.
- Probe paths are diagnostics only; they do not describe shipping behavior.
- Promotion evidence comes from direct frontier-vs-baseline duels, not fixture churn alone.
- `runtime-preflight` still matters after promotion: exact-lite is hard, stage-1 CPU is advisory for Pro.
- Automove users can play any current `GameVariant`; promotion evidence must cover variant breadth, not just Classic.

## Experiment Rules

- Pick one hypothesis before editing runtime code.
- Probe first when the mechanism is unclear. Do not spend canonical gates on a guess.
- When the live board has no challenger, use the structural reset in `docs/automove-strategy.md`; do not continue the seam-fix loop by default.
- Run `./scripts/run-automove-structural-scout.sh <candidate>` before retaining any broad Pro runtime change. A candidate must look strong on both canonical sampled and active-blocker panels, not just one.
- Treat retained Classic fixtures as regression controls, not proof that a candidate is stronger across variants.
- Use seeded sampled variants for quick kill/pass evidence, then all-variant confirmation for promotion.
- Separate `pre_accept` search choice from final `engine_post_search` output before changing shared heuristics.
- A seam can move while the duel gate still fails; local seam coverage is not duel strength.
- Use `SMART_PRO_RELIABILITY_HOTSPOT_FEN` on `smart_automove_pro_reliability_hotspot_probe` for one-off board inspection. Do not add temporary source cases just to inspect a copied trace board.
- Use `SMART_PRO_RELIABILITY_DUEL_FILTER` on `smart_automove_pro_reliability_duel_trace_probe` or `smart_automove_pro_reliability_nonwin_trace_probe` when only one duel bucket needs recurrence evidence.
- Use `smart_automove_pro_profile_sweep_probe` when the live surface is too mixed for another seam patch. The probe compares test-only Pro candidates without adding them to the retained profile registry; its output is discovery evidence, not promotion evidence.
- Prefer the `run-automove-experiment.sh` diagnostic stages for repeatable profile comparisons: `pro-profile-sweep` wraps `smart_automove_pro_profile_sweep_probe`, and `pro-profile-attribution` wraps `smart_automove_pro_profile_attribution_probe`. These stages are diagnostics only; retained-profile restrictions still apply to canonical promotion stages.
- Use `pro-promotion-dashboard` before spending runtime code on a broad Pro candidate. It reports sampled plus active-blocker shape, weakest variant rows, candidate-vs-shipping strength, and candidate-vs-guarded pressure in one run; profile attribution should follow only when the dashboard leaves a globally promising candidate with one explainable miss. For high-risk or expensive scouts, first run `SMART_PRO_DASHBOARD_PANEL_FILTER=sampled SMART_PRO_DASHBOARD_INCLUDE_GUARDED=false` and kill immediately on a sampled miss.
- Use `pro-policy-matrix` when candidate policies are complementary on isolated active rows but suspect across sampled or mode panels. The first candidate is the baseline; the matrix replays identical openings and reports candidate-vs-baseline outcome deltas, nonwins, first-move divergences, and branch/context/pair aggregates.
- Use `pro-policy-winner` after a policy matrix has good oracle coverage and you need selector design data faster. It short-circuits candidate replay after the first winning policy for each baseline loss and aggregates the winner policy, exact context, and move pair.
- Broad `pro-policy-winner` portfolios are still expensive. Prefer `SMART_PRO_POLICY_WINNER_PANEL_FILTER`, `SMART_PRO_POLICY_WINNER_DUEL_FILTER`, and `SMART_PRO_POLICY_WINNER_CANDIDATE_TRACE_LIMIT` when refreshing coverage; treat `candidate_trace_limit_hit=true` summaries as partial diagnostic evidence only.
- Treat `pro-policy-winner` output as selector evidence, not promotion evidence. The first winning policy can be a later manifestation of a policy that needed to enter earlier, and the same exact-opportunity context can prefer different policies across Pro/Normal/Fast; always validate any selector on `pro-promotion-dashboard`.
- Sampled Pro policy-winner coverage alone is a false positive. A selector that patched the current sampled Pro winner contexts reached `11-1` against shipping Pro, then failed sampled Normal at `8-4` and sampled Fast at `6-6`. The later `corner_chain_mana_rows` white `no_policy_win` was a forced-root oracle harness bug, not a root-pool ceiling, but fixing that coverage did not make static selectors promotable.
- Full rollout policy selectors are not a practical escape hatch. A test-only portfolio rollout from early `frontier_execute` boards with only 24 internal plies did not complete a sampled Pro quick check after several minutes, so future rollout ideas need a much cheaper feature extraction path before they can be promotion candidates.
- Use `smart_automove_pro_decision_record_aggregation_probe` before building a shipping-root or context meta-selector. Run `SMART_PRO_DECISION_RECORD_SCOPE=nonwins` when the missed promotion gate contains flat losses, because delta-only attribution hides games where both frontier and shipping lose.
- Use `smart_automove_pro_forced_root_oracle_probe` when a policy matrix class has `no_policy_wins` and you need to know whether winning roots are already present below the selected root. The probe is a root-set diagnostic; it does not justify a direct profile unless the resulting policy also survives sampled and active dashboard panels.
- Preserve case for env-supplied FENs and other payloads. `SMART_PRO_FORCED_ROOT_ORACLE_FEN` exposed that the canonicalizing env string helper lowercases values; lowercased FENs can turn a real root-selector miss into a false zero-root no-go.
- When forcing roots from a board copied out of a full-opening first-divergence record, set `SMART_PRO_FORCED_ROOT_ORACLE_START_PLY` to that record's `first_diff_ply`. Without this, the oracle gives the copied board a fresh full `SMART_PRO_FORCED_ROOT_ORACLE_MAX_PLIES` horizon instead of the remaining full-opening horizon, inflating local wins. The active white `outer_edge_mana_rows` Pro/Fast boards changed from `12/16` and `15/16` unadjusted wins to corrected `8/16` and `9/16`; the guarded-selected roots then matched the full-opening losses.
- Use `PRO_POLICY_MATRIX_PORTFOLIO` and `PRO_POLICY_MATRIX_PORTFOLIO_CLASS` before retaining a selector over existing policies. `candidate_only_wins` is available upside, `baseline_only_wins` is the save/regression risk, and `no_policy_wins` means the current policy set lacks a winning line for that slice.
- A complete policy portfolio is not enough when the required policy entry point differs from the printed first divergence. One selector reached sampled Pro `11-1`, Normal `12-0`, Fast `11-1`, then failed active blockers at Pro `4-2`, Normal `6-0`, Fast `5-1`; after adding the recovered white-opening utility policy, the sampled dashboard reached `11-1 / 11-1 / 11-1` but active Fast still failed at `5-1`. Do not retry static turn/color/`mons_moves` policy gates on that evidence alone.
- The active `outer_edge_mana_rows` white turn-three `window=0/deny=0/drainer_safety=2` context is not one-sided. A wider outer-edge delta run found the same context as a Pro improvement on one opening and a Fast regression on another, and also exposed a separate Pro black regression. A selector keyed by guarded-selected move or board FEN would be a dashboard memorization, not a promotable automove feature.
- Use the enriched `PRO_PROFILE_SWEEP_ATTRIBUTION_CONTEXT` output before building any context-gated meta-selector. A branch name is too coarse; require variant, color, turn, `mons_moves`, action/mana availability, and exact-opportunity context to be one-sided across Pro, Normal, and Fast before retaining a gate.
- Baseline-root liveness is not enough for a shipping-root meta-selector. A decision-record wave found the shipping root candidate-live, advisor-ordered, or advisor-approved in several nonwins, but the exact mechanisms still split across rank-0 same-family mana drift, outside-shortlist recovery approval, safe-progress-vs-spirit approval, accepted-head spirit drift, and score-window fallback. A test-only live-root shipping fallback then failed the active Pro panel: rank-0 shipping fell to `1-5`, while top-2/advisor-live variants stayed at `2-4`.
- Shallow ProV3 utility root switches are not enough by themselves. Pure utility root selection failed active blockers, and the best candidate-set nonstrict utility switch only reached active blockers `5-1 / 5-1 / 5-1` before canonical sampled Pro failed at `5-7`. Score/eval tolerance tweaks and a local early-white quiet-turn guard worsened Pro, so do not retry root-level utility ordering without new utility features or dashboard evidence that it is strong on sampled and active panels together.
- Utility-ordering the entire reply-risk shortlist is also too blunt. A full utility-ordered reply-risk scout changed almost every turn, failed sampled Pro at `3-9`, zeroed `forward_bridge_mana_rows`, and raised average candidate move time to `284.52ms`; future utility work needs a new feature or narrower mechanism, not broad utility-first ordering.
- Policy-output utility filtering is not a practical shortcut around the killed portfolio selectors. A test-only scout that evaluated known policy outputs and admitted only moves whose `TurnEngineUtility` cleared guarded override/eval gates did not finish a sampled Pro-only dashboard after several minutes, so it is non-promotable on cost before strength.
- Dormant ProV2 config toggles are not hidden promotion switches. Active-blocker dashboard runs killed eligibility gating, selected-followup projection off, secondary analysis off, late-safe-mana preference off, late-black setup rescue, and bounded root injection; all failed at least Pro or Fast before sampled validation.
- Generic ProV3 head/root utility floors are too blunt. Requiring a head or scored root to support the selected root within eval tolerances `96` or `192` collapsed active Pro to `2-4` with `outer_edge_mana_rows` at `0-2`; future ProV3 work needs new utility features, not a generic post-search head floor.
- Root-level shipping/utility consensus is still too shallow. A test-only selector that preserved guarded fallbacks and only let a candidate-live, reply-safe shipping root override `frontier_execute` failed active Fast at `2-4`, with `alternating_mana_rows` `0-2`. The nearby no-post-search-head selector also stayed non-promotable at active Fast `3-3`, so do not retry this family without a new utility feature below head/pre-accept selection.
- Selective reply-risk head rejection and quiet spirit-representative promotion are also too shallow. Rejecting only reply-risk-dominated non-concrete heads stayed at active Fast `3-3`; adding safe quiet `SpiritImpact` representative promotion over non-concrete `ManaTempo` also stayed `3-3`, even with loose rank/score/reply tolerances. The residual still split across early-white preserved-spirit, late-white same-family head drift, and black flat-loss surfaces.
- Discard context-gated meta candidates that improve only sampled Pro while rotating Normal/Fast. One structural wave improved sampled Pro-vs-Pro from `7-5` to `10-2`, but broad/raw gates collapsed Normal/Fast, portfolio selectors failed Pro, and the exact gate still missed `alternating_mana_rows` plus `forward_bridge_mana_rows`; that is evidence for deeper decision-record aggregation or ProV3 utility work, not another branch-context toggle.
- Even attribution-informed exact context gates can remain too shallow. A second exact-context probe improved isolated alternating/black unsafe contexts but still failed structural scout: sampled `8-4 / 8-4 / 6-6` and active blockers `3-3 / 5-1 / 3-3` across Pro/Normal/Fast. Do not retry opening or late-black context switches without a deeper decision record showing a repeated advisor/head/final-selection mechanism.
- Passing the small `pro-reliability` gate does not guarantee confirm readiness.
- Passing sampled variants does not guarantee all-variant readiness.
- Stacked narrow late-ply head/advisor overrides are especially suspect. One direct line cleared sampled `pro-reliability` at `1.0000 / 0.9167 / 0.9167` and then collapsed in all-variant confirm at `0.6667 / 0.7292 / 0.6667`.
- Treat rotated residue from a discarded challenger as provisional until a clean-tree retained trace confirms it.
- Do not promote a live `first_diff_ply` board into retained until the retained replay reproduces the same final shipping-selected root; copied late-ply snapshots can collapse back to frontier on clean replay.
- Runtime cost is a real gate. A candidate that wins local seams while drifting further into the `1.5x+` advisory band is not an upgrade.
- Wrapper-only reroutes, fallback widening, shortlist widening, and metadata-only advisor changes saturate quickly; durable progress usually needs shared approval, head, or scoring logic.
- Extending a retained shipping-fallback path to a wider black weak-window surface is not automatically safer than advisor surgery; one direct late-black shipping-fallback expansion fixed the traced sampled boards and still rotated the sampled gate down to Fast `0.7500`.
- When an explicit-variant nonwin trace splits the remaining sampled residue across different turn families or selector stages, keep the wave diagnostic-only. The current `outer_edge_mana_rows` plus Fast `forward_bridge_mana_rows` / `corner_chain_mana_rows` residue did not collapse to one mechanism, and isolated Fast `classic` did not reproduce.
- A widened recurrence trace can still fail to justify runtime code even when one pair repeats. On `alternating_mana_rows,forward_bridge_mana_rows`, the only repeated Fast pair was the archived white head-accept seam `l9,6;l7,4;l7,3` vs `l9,6;l7,6;l7,7`, while every other Fast regression pair stayed singleton; that is still mixed residue, not a shared runtime hypothesis.
- A refreshed explicit-variant trace that reproduces an archived mixed bucket is not new promotion evidence. The `outer_edge_mana_rows` Normal refresh again logged `10` nonwins with only black `l2,7;l3,8` vs shipping `l1,5;l0,3;l1,3` repeating, while the rest stayed split across pre-accept, accepted-head, copied-board, and forced-prepass surfaces; another special-case head guard would not address that breadth.
- Refreshing the combined Fast blockers can reproduce the same no-go without creating a new spend. The `alternating_mana_rows,forward_bridge_mana_rows` Fast refresh logged `9` nonwins with only the archived white `forward_bridge` accepted-head pair repeating; the remaining singleton misses matched prior copied-board or retained-extension no-go classes.
- A canonical sampled gate refresh is useful for blocker truth, but it is not a runtime hypothesis by itself. The latest clean-tree gate still failed only on Normal `outer_edge_mana_rows` and Fast `alternating_mana_rows` / `forward_bridge_mana_rows`; if targeted traces for those variants remain mixed, do not spend code just because the overall gate reproduced the same failure.
- A clean sampled gate refresh after focused singleton-only traces is useful as promotion truth, not as a code spend. One refresh after the alt-v5 focused Normal/Fast no-gos still failed at Normal `0.9167` confidence `0.9968` and Fast `0.8333` confidence `0.9807`, with the same weak rows: Normal `outer_edge_mana_rows`, Fast `alternating_mana_rows`, and Fast `forward_bridge_mana_rows`.
- A nonwin trace can still be too broad for code even when it targets the exact failed Fast rows. On `alternating_mana_rows,forward_bridge_mana_rows` with default sampled seed and `vs_shipping_fast`, the trace logged five nonwins across black bridge/mana, black setup, black late recovery, white safe-progress, and white head-accept surfaces, with no repeated move pair.
- A nonwin trace can also be too small and split for code. On default-seed `outer_edge_mana_rows` with `vs_shipping_normal`, the two frontier nonwins were different surfaces and both were losses for shipping as well, so they did not identify a direct Normal repair.
- A white `forward_bridge_mana_rows` same-setup `SpiritImpact` head reject is a sampled-only repair, not a promotion path. Both broad and exact-shape versions removed the sampled Fast head-accept regression and passed sampled `pro-reliability` at `1.0000 / 0.9167 / 0.9167`, then failed all-variant confirm at `0.6667 / 0.7292 / 0.6667`.

The variant-scoped cut again passed sampled `pro-reliability` at `1.0000 / 0.9167 / 0.9167`, then failed all-variant confirm at `0.6667 / 0.7292 / 0.6667` with broad per-variant floor failures.
- A root-advisor trace refresh is not enough for code when the boards split across approval and injection surfaces. The current blocker-adjacent cases used `ApprovedReplyRiskGuard`, `ApprovedFamilyCompetition`, omitted-root reentry, preserved representatives, and rejected injected macro roots; that diversity kills broad advisor-label, reentry, representative, or injection patches.

It removed the repeated `alternating_mana_rows` `l2,7;l1,6` miss in the isolated Fast trace, but sampled reliability still failed and rotated Normal lower; local removal of one outside-shortlist recovery jump is not multi-variant strength.

One direct fallback passed early stages but left sampled `pro-reliability` unchanged at `1.0000 / 0.9167 / 0.8333`; do not retry it without a new trace proving it moves the live blockers.

Enabling head rerank, disabling spirit-family expansion, disabling the mid-turn tactical guard, raising only `turn_engine_expansion_cap`, and disabling the low-budget guard all passed early gates and still failed sampled `pro-reliability`; the low-budget and spirit-family removals also broadened Pro losses.

A test-only scout found most guarded frontier roots were still same-turn states rather than post-handoff states (`428` pressure-missing turns on sampled Pro), produced no overrides, and reproduced the guarded `7-5` sampled-Pro miss.

Do not reopen completed-plan pressure/admission without a new utility feature or evidence that candidates survive the score/safety/utility gates.
- Preserving the turn-engine continuation cache is a real structural lever, but not a direct challenger. A persistent-plan-cache guarded scout improved sampled Pro only to `8-4`; adding the retained opening components reached `9-3`; adding raw ProV2 on `inner_wedge_mana_rows` reached `10-2`; widening raw to `forward_bridge_mana_rows` rotated the row and fell back to `8-4`. Disabling `enable_turn_engine_low_budget_guard` only for `ForwardBridgeManaRows` passed sampled `pro-reliability` at `1.0000 / 0.9167 / 0.9167` and moved sampled Fast `forward_bridge_mana_rows` to `1.0000`, then failed all-variant confirm at `0.6458 / 0.7083 / 0.6667` with broad per-variant floor failures.

## Retained Package Lessons

- The promoted retained package is a set of narrow, mutually constrained repairs around `frontier_pro_v2_guarded`; keep them together unless a future gate proves otherwise.
- White turn-three and turn-five repairs are intentionally split by mechanism: no-action recovery, selected-rank search-only fallback, equal-surface ProV1 tiebreaks, and final head rejection are not interchangeable.
- Black late setup/progress repairs belong in advisor family competition only when setup roots are close-surface, already shortlisted, and backed by retained controls.
- A direct black progress-vs-setup wrapper mirror is unsafe: it fixed one local board and failed confirm by rotating onto later Fast seams.
- A broad white turn-three recovery or raw ProV1 search-only reroute is unsafe: it fixes known siblings but flips retained vulnerable guards or rotates Normal losses.
- A root-rank separator is not enough for white search-order fallbacks; selected rank was the safe separator for the retained negative-deny sibling.
